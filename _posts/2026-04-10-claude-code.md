从 2024 年开始，我就在大量使用 AI 辅助开发。早期主要是让它补全代码片段、解释报错、写单元测试——有用，但谈不上颠覆性。

真正让我感受到质变的，是 Claude Code 出来之后。

---

## 一、背景：为什么对全栈开发者来说意义更大

我是全栈开发，Android、iOS、后端、前端管理台都要写。以前一个人做项目，时间分配是最大的瓶颈——写完后端写前端，前端写完再切到 Android，iOS 写完还要做管理后台的交互，每次切换技术栈都要重新调状态，效率很难提上去。

虽然一个人能顶一个小型团队，但时间是有限的，很多想做的项目因为时间成本太高就搁置了。

Claude Code 出现之后，这个局面发生了变化。它不只是补全代码，而是真正能理解项目上下文、按照架构规范生成代码、在不同技术栈之间自如切换。对全栈开发者来说，这比只擅长单一语言的开发者收益更大，因为每个技术方向都能被加速。

---

## 二、和 Cursor 的对比

在用 Claude Code 之前，我用过一段时间 Cursor。用下来的感受是——**不太理想**，主要问题集中在几点：

**上下文理解不够深**。Cursor 对当前文件的理解不错，但对整个项目的架构感知比较弱。跨文件的修改经常需要我手动指定文件，它很难主动理解"这个改动会影响哪些地方"。

**生成的代码风格不一致**。同一个项目里，不同时候生成的代码命名规范、错误处理方式、注释风格各不相同，后期维护成本高。

**对话上下文容易丢失**。稍微复杂一点的需求，聊几轮之后它就开始"忘记"之前说过的约定，需要反复强调。

Claude Code 在这几个方面明显更好，尤其是项目级的上下文理解和代码风格一致性。

## 三、Claude Code 的工作方式

### 3.1 CLAUDE.md 架构文档驱动

这是我认为 Claude Code 最核心的用法。在项目根目录创建 `CLAUDE.md`，把项目的架构设计、技术栈选型、命名规范、模块划分、接口约定全部写进去：

```markdown
# 项目架构说明

## 技术栈
- Android：Kotlin + Jetpack Compose + MVVM
- iOS：Swift + SwiftUI + MVVM
- 后端：Spring Boot + Spring Cloud Alibaba
- 前端管理台：Vue3 + TypeScript + Element Plus

## 命名规范
- 接口统一返回 Result<T> 包装
- 错误码统一在 ErrorCode.kt / ErrorCode.swift 中定义
- Repository 层负责数据源，ViewModel 层负责业务逻辑

## 模块划分
- user-service：用户相关
- order-service：订单相关
- api-gateway：统一网关入口

## 注意事项
- 所有网络请求必须有超时处理
- 敏感操作需要二次确认弹窗
```

有了这个文件，Claude Code 每次生成的代码都会遵循这套规范，不需要我反复强调。这解决了 Cursor 代码风格不一致的问题。

### 3.2 子任务切分

复杂功能我会先拆分，然后一个任务一个任务地让它完成：

```
任务：实现用户订单列表页面（Android）

子任务：
1. 创建 OrderListViewModel，包含分页加载逻辑
2. 创建 OrderListScreen（Compose），实现列表 UI 和下拉刷新
3. 创建 OrderRepository，对接后端接口
4. 编写单元测试
```

每个子任务单独执行，完成后 review 一遍再继续下一个。比一次性说"帮我做一个订单列表"效果好很多，因为每一步都在可控范围内。

### 3.3 结合 Git 工作流

我的习惯是：

1. 开始一个新功能前，先在 `CLAUDE.md` 里补充这个功能的设计说明
2. 让 Claude Code 生成代码框架
3. 自己 review 并补充业务细节
4. `git commit`，进入下一个子任务

这样每个 commit 都有对应的 Claude Code 生成记录，方便后续排查问题。出了 Bug 也能快速定位是哪个阶段引入的。

### 3.4 直接说需求，让它写

对于标准化程度高的代码（CRUD、工具类、数据模型、接口定义），我直接描述需求：

> 帮我写一个 Swift 的网络请求封装类，基于 async/await，支持 GET/POST，统一处理错误，返回 Result 类型，超时时间 10 秒。

这类代码 Claude Code 生成质量很高，基本不需要大改，直接用。省下来的时间用在真正需要思考的业务逻辑上。

## 四、最让我惊喜的一次体验

有一次我在做一个后端接口，写到一半发现数据库设计有问题——一个字段的类型设计不合理，影响到三个相关服务。

以前遇到这种情况，我需要：手动找出所有受影响的文件、逐个修改、确认改动没有遗漏、更新相关测试。一般要花一两个小时。

这次我直接告诉 Claude Code：

> user 表的 status 字段从 Int 改成 String 枚举类型，值为 "active"/"inactive"/"banned"，帮我找出项目里所有用到这个字段的地方，统一修改，包括数据库 migration、DTO、业务逻辑判断、接口文档注释。

它扫描了整个项目，列出了 12 处需要修改的地方，然后逐一修改，最后生成了一个变更摘要让我 review。整个过程不到十分钟。

这种跨文件、跨模块的联动修改，是我觉得 Claude Code 和普通 AI 辅助工具差距最大的地方。

## 五、Claude Code 结合 OpenClaw 使用

两者结合之后，形成了一套"随时随地开发"的工作流：

**场景：不在电脑旁，需要让 Claude Code 继续推进任务**

我在飞书给 OpenClaw 发消息：

> 打开终端，进入 ~/projects/my-app，让 Claude Code 继续完成昨天没写完的用户登录模块，按照 CLAUDE.md 的规范，生成 LoginViewModel 和 LoginScreen，完成后 git commit。

OpenClaw 会在我的 Mac 上执行这个操作，启动 Claude Code、传入任务描述、等待完成后执行 git commit，最后把结果发回给我。

**场景：快速验证一个想法**

在外面突然想到一个功能设计，不用等回家：

> 帮我在 ~/projects/my-app/docs 下创建一个新文件 feature-idea.md，内容是：[功能描述]，然后让 Claude Code 读取这个文件，生成一个初步的技术方案，保存到同目录下的 feature-idea-impl.md。

整个想法的捕获和初步实现，在手机上发几条消息就完成了。

## 六、总结

Claude Code 改变了我对"一个人做项目"这件事的预期上限。

以前我的瓶颈是时间——技术栈多，每个方向都要花时间，一个人很难同时推进多条线。现在标准化的代码生成交给 Claude Code，我的时间集中在架构设计、业务逻辑和 review 上，效率差距非常明显。

几个关键经验总结：

**CLAUDE.md 是核心**。把架构文档写好，代码质量和一致性会有质的提升。这份文档投入的时间，会在后续每次生成中回收回来。

**子任务切分比一次性描述效果好**。大需求拆小，每步可控，review 成本低，也更容易发现问题。

**结合 OpenClaw 的组合**让开发从"坐在电脑前才能进行"变成了"随时随地可以推进"，对我这种经常不在电脑前的开发习惯非常契合。

目前我自己的几个项目推进速度比以前快了很多。之前一个人做全栈是迫不得已，现在感觉更像是一种优势——每个方向都能被 AI 加速，覆盖面越广，收益越大。
