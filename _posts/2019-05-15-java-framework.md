> 从单机到集群，从 SOA 到微服务——每一次演进，都是业务规模倒逼的结果。

## 单机应用

最早期的架构形态，所有业务逻辑、数据库、静态资源全部部署在一台服务器上。为了保证基本的高可用，通常会引入 LVS + Keepalive 做故障切换。

单体应用的局限随着业务增长逐渐暴露：开发速度越来越慢、启动时间越来越长、模块之间的依赖越来越难以梳理。

![单机应用架构](/assets/images/2019/2019051501.png)

## 单体集群架构

将同一个单体应用部署到多台服务器，通过负载均衡统一对外提供服务。这一阶段主要解决了三个问题：

**高可用**：只要集群中有一个节点存活，服务就能正常对外响应。通过将故障节点上的流量自动切换到其他节点，系统正常运行时间可以达到 99.9% 以上。

**吞吐量**：横向扩展节点数量，并发处理能力随之提升，能够支撑更大规模的用户访问。

**易扩展**：在预判流量增长时，可以提前增加节点，弹性伸缩，而无需改动应用本身。

![集群架构](/assets/images/2019/2019051502.png)

## SOA 架构

SOA（面向服务架构）的核心思路是按照业务边界对单体应用进行垂直拆分，拆分后的各个服务之间通过消息总线（ESB）进行交互。每个服务内部仍然是单体结构。

SOA 与微服务在思想上有交集，但侧重点不同：

- **SOA**：强调服务的拆分与接口访问的统一化，ESB 是核心枢纽；
- **微服务**：强调将公共能力抽取为独立服务，去中心化，服务间直接通信。

可以说，SOA 架构中已经包含了微服务的部分思想，微服务是对 SOA 的进一步演进和轻量化。

![SOA 架构](/assets/images/2019/2019051506.png)

## 微服务架构

微服务将每个业务能力拆分为独立部署、独立运行的小服务，每个服务职责单一，易于开发、理解和维护，也可以独立启动和发布。

**主要优势：**
- 服务边界清晰，团队可以独立开发和部署；
- 单个服务故障不影响整体系统；
- 技术栈灵活，不同服务可以按需选型。

**主要挑战：**
- 分布式系统带来的事务一致性问题；
- 服务数量增多后的治理复杂度上升。

![微服务架构](/assets/images/2019/2019051503.png)

### 微服务核心组件

微服务体系中有几个关键的基础设施，缺一不可：

**网关**：统一入口，负责路由转发与请求过滤。例如：
- `/api/v1/product/` → 商品服务
- `/api/v1/order/` → 订单服务
- `/api/v1/user/` → 用户服务

**服务注册与发现**：维护服务调用方与被调用方的地址信息，实现动态寻址。

**配置中心**：集中管理各服务配置，支持动态刷新，无需重启服务。

**链路追踪**：分析跨服务调用的耗时与依赖关系。例如一次下单请求，背后可能涉及：查询商品价格 → 查询用户信息 → 写入订单数据库，链路追踪能清晰还原每一跳的耗时。

**负载均衡**：将请求分发到多个服务实例，避免单点压力过大。

**熔断**：当被调用方出现异常时，主动断开调用链路，保护自身不被拖垮，同时给下游恢复时间。

![微服务核心组件](/assets/images/2019/2019051504.png)

### 常见微服务框架

微服务架构中，服务分为调用方（consumer）和被调用方（provider），同一个服务在不同场景下可以同时扮演两种角色。

目前主流的两个框架体系：

**Dubbo**（`zookeeper + dubbo + Spring Boot`）
- 通信方式：RPC
- 注册中心：ZooKeeper / Redis
- 配置中心：Diamond
- 官网：[dubbo.apache.org](http://dubbo.apache.org)

**Spring Cloud**（全家桶，易于集成第三方组件）
- 通信方式：HTTP RESTful
- 注册中心：Eureka / Consul
- 配置中心：Spring Cloud Config
- 熔断：Hystrix
- 网关：Zuul
- 链路追踪：Sleuth + Zipkin
- 官网：[projects.spring.io/spring-cloud](http://projects.spring.io/spring-cloud/)

![常见微服务框架对比](/assets/images/2019/2019051505.png)

## 实战：微服务电商项目基础模块设计

以一个小而精的电商项目为例，演示微服务的模块拆分方式：

**用户服务**
- 用户信息查询接口
- 登录接口

**商品服务**
- 商品列表
- 商品详情

**订单服务**
- 我的订单
- 下单接口

三个服务职责清晰、边界分明，是学习微服务落地实践的良好起点。
