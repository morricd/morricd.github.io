## 关于工作
2024 年的工作，用两个字概括：失业。
但如果要用完整的说法，应该是：失业，然后比以前更忙。
年初离职，原本打算休息一阵，顺带找找方向。结果方向没找到，倒是接了一个独立开发的项目——一个跨端业务平台，iOS 和 Android 双端，从零开始搭。
接下来的八个月，我一个人写了将近五十万行代码。
说出来有点荒唐。但这是真实发生的事。
客户管理、会议日历、课程、视频、直播、商城、钱包——这些业务模块，我一个一个拆解，一个一个落地，iOS 用 Swift，Android 用 Kotlin，两端并行，需求频繁变动，版本持续发布。
2023 年我在总结里写过：学了很多，用出来的很少。
2024 年这个问题算是被强行解决了——不是因为我想通了，是因为没有退路。
最难的不是代码量，是在需求不断变化的情况下，维持双端的逻辑一致，还得保证每次上线不崩。单人项目没有 code review，没有人兜底，出了问题只能自己查。这种压力，和在公司里做螺丝钉是完全不同的感受。
有没有成就感？有。
有没有很累？也有。
值不值得？还没想清楚。

### 简聊一下这个项目的后端
这个项目的后端我没有参与，由几位初级 Java 开发和从 .NET 转来的同事负责。整体架构没有采用微服务体系，只是简单挂了一个 Gateway 网关做路由，其余的服务治理、熔断、限流、监控一概没有。

问题集中在设计层面：客户表搞了两套，支付逻辑图省事怎么快怎么来，接口没有统一的返回格式，身份认证也是各做各的。直接影响是对接成本翻倍——一个页面要请求两个接口才能确认用户身份，数据还是分散的两份，很多本可以避免的冗余开发因此堆出来。

刚入职那段时间，这些问题确实让人憋屈。但自己不是管理层，团队里也没有人负责技术规范和架构把控，只能在自己负责的移动端部分尽量做好。

说到底，外包项目的逻辑就是快，质量往往排在后面，这是这类项目的结构性矛盾。

不过这种落差，也让我更清楚地看到自己之前在后端架构上积累的东西是有价值的。

我在中大型互联网项目中先后用过 Spring Cloud Netflix 和 Spring Cloud Alibaba 两套微服务体系。Netflix 那套走的早，Eureka、Hystrix、Zuul 都上过生产；后来随着这些组件逐渐停止维护，切换到了 Spring Cloud Alibaba——Nacos 做注册中心和配置中心，Sentinel 做限流熔断，Gateway 替换 Zuul。两套都在实际项目里跑过，对各自的优劣有直接的认识，不是只停留在概念层面。

自己主导后端架构时，通常会从以下几个方向建立规范：

接口规范：统一 RESTful 风格，所有接口返回格式一致，错误码体系统一定义，前后端对接成本明显降低；
数据库设计：表结构命名规范、字段类型约定、分库分表策略提前规划，避免业务增长后改表的噩梦；
身份认证与权限：JWT + RBAC 权限模型，统一鉴权入口，不让各服务自己搞一套；
服务治理：熔断、限流、降级策略在上线前就配好，不等线上出事再补；链路追踪接入 Sleuth + Zipkin，出了问题有据可查；
CI/CD 与部署：Jenkins 或 GitLab CI 自动化构建，Docker 容器化，环境隔离做清楚，减少"本地好好的，上线就挂"这类问题。

这些东西不是复杂的技术，更多是工程习惯。有没有这套习惯，做出来的系统差距会很大——在这个外包项目里，我对这一点感受尤为直接。

## 关于学习
这一年工作强度高，从去年 9 月入职到现在，周末基本都扑在项目开发上，系统性学习几乎停了下来。

但完全不学也做不到。零碎的时间里断断续续看了一些——有时候是项目里遇到的技术点顺手深入查一查，有时候是睡前刷一段视频或者翻几页书。碎片化学习效率不高，知识很难成体系，但总比完全停下来要好。

这种状态让我有些不安。学习这件事一旦中断习惯，重新捡起来就需要更大的力气。等这个项目阶段性收尾之后，我需要把学习节奏重新拉回来。

## 关于生活
成都的日子，说不上有什么大起伏，就是普通的日常。

工作日上班写代码，下班回来有时候继续写，周末也多半对着电脑。成都的生活成本不算高，节奏也没有北上广那么逼人，这是让人住得下去的地方。偶尔周末在附近转转，算是给高密度工作状态里留一点空隙。

生活简单，有时候也会觉得单调。但简单也有简单的好处——没有太多干扰，想做什么就做什么。
