## 之前没有总结，从现在起每年都总结自己的学习和生活

时间一天天溜走，回头看才意识到，自己竟从未做过一次认真的年终总结。工作和生活就这样在忙碌中漂移，缺乏节奏，也缺少沉淀。因此，从 2019 年起，我决定无论多忙，都要在年底为自己留下一份记录——回顾工作、梳理成长，让每一年的努力都有迹可循。

---

## 技术成长：从单体架构到微服务的完整蜕变

这一年，是我在系统架构认知上发生根本性转变的一年。

过去参与的项目大多基于**单体架构**：所有业务逻辑耦合在一个工程里，部署简单，但随着公司业务快速扩张，扩展性瓶颈、稳定性问题接连暴露，单体的局限越来越明显。于是，我开始主导推动项目向 **SOA → 微服务**方向演进。

### 第一阶段：Spring Cloud Netflix 体系的学习与实践

在微服务改造初期，我系统学习了 Spring Boot 与 Spring Cloud，并围绕 Netflix 技术栈构建了一套完整的服务治理体系：

| 组件                | 职责                     |
| ------------------- | ------------------------ |
| Zookeeper / Eureka  | 服务注册与发现           |
| Dubbo               | RPC 服务调用框架         |
| Hystrix             | 熔断、降级、限流         |
| Zuul                | API 网关，统一路由与鉴权 |
| Spring Cloud Config | 分布式配置管理           |

第一遍学习视频时，很多内容只有模糊印象；第二遍结合 demo 动手实践，才真正理解每个组件在架构中的定位与职责。**理解架构，必须自己搭过一遍。**

### 面对的核心挑战

微服务改造并非一帆风顺，主要踩坑集中在以下几个方向：

**① 分布式事务**：多个服务共同完成一个业务操作，数据一致性极难保障。尝试过 TCC 模式与最终一致性方案，深刻理解了"网络不可靠"这一前提对系统设计的影响。

**② 服务治理**：调用链路的可观测性、熔断阈值的调优、配置的动态下发，每一项都需要额外的工程投入，绝非引入一个组件就能解决。

**③ 分库分表**：为了配合服务拆分做数据层改造，啃了多本专业书籍，也查阅了大量实战博客，逐步摸索出适合公司场景的分片策略。

**④ 资源受限**：公司仅有 3～6 台服务器，Docker 环境搭建、服务部署经常遇到资源竞争，加班处理突发状况成了家常便饭。在资源受限的条件下做架构，也是一种非常宝贵的历练。

### 第二阶段：转向 Spring Cloud Alibaba

2019 年中，我参加了**华为和阿里巴巴的成都技术峰会**，这两场会议开阔了我的视野，也让我意识到 Netflix OSS 生态的隐患：

- Eureka 2.x 停止开发，社区活跃度持续下降；
- Hystrix 进入维护模式，官方建议迁移；
- Zuul 1.x 基于阻塞 IO，高并发场景性能有限；
- 整体配置繁琐，缺乏可视化管理界面。

因此，我开始系统学习阿里巴巴的微服务解决方案：

| 组件                 | 替代的 Netflix 组件 | 核心优势                                             |
| -------------------- | ------------------- | ---------------------------------------------------- |
| Nacos                | Eureka + Config     | 注册中心与配置中心二合一，支持动态刷新与可视化控制台 |
| Sentinel             | Hystrix             | 流量控制更精细，实时监控面板直观，规则动态推送       |
| Spring Cloud Gateway | Zuul                | 基于 Reactor 非阻塞异步模型，性能大幅提升            |

由于前期 Netflix 体系打下的基础，学习阿里组件时上手很快——核心思想是相通的，差异主要在实现细节和运维体验上。到年底，基于 Spring Cloud Alibaba 的一套微服务体系已在公司项目中逐步落地。

### 补强 JVM 底层

除了架构层面的演进，这一年我也系统复习了 **JVM 运行机制**，包括：

- 类加载机制与双亲委派模型
- 内存模型（堆、方法区、栈的划分）
- 垃圾回收算法（G1、CMS 的使用场景与调优参数）
- 常见 OOM 类型的排查思路

这些底层知识在排查线上问题时发挥了直接作用，也让我对整个 Java 体系的理解更加立体。

## 数学基础的重拾：为 AI 方向做准备

早在 2017 年，我买过一本《深度学习与计算机视觉》，但翻开第一章就被线性代数和概率论劝退了——大学学过，早已还给老师。

2019 年，我下定决心系统补课：

- **高等数学**：重新理解极限、导数、梯度——这些才是反向传播算法的语言；
- **线性代数**：矩阵运算、特征值分解，深度学习的每一步计算都绕不开；
- **概率论与统计**：贝叶斯定理、期望与方差，理解模型输出的前提；

过程确实痛苦，尤其是重新找回抽象符号推导的感觉。但我深知，**不懂数学，就只能调调参数，永远停在工具的表面。** 这一年的复习，让我再次拿起《深度学习与计算机视觉》时，终于能跟上作者的推导节奏了。

## 总结

2019 年，是我从"会用框架"走向"理解架构"的一年，也是在悲伤中咬牙坚持成长的一年。

微服务体系从 Netflix 到 Alibaba 的迁移，让我理解了技术选型背后的权衡逻辑；分布式事务与服务治理的踩坑，让我意识到架构设计从来不是纸上谈兵；数学基础的重拾，则是我向更深层技术方向迈出的第一步。

生活给了我一些考验，技术给了我一个出口。

**未来每一年，不论多忙，都要写下来。记录技术，记录思考，记录成长——让每一年都不被辜负。**
